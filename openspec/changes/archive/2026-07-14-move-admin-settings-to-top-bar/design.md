# Design: move-admin-settings-to-top-bar

## Context

В `AdminScreen.tsx` компонент `SettingsScreen` (сегментед-контролы выбора AI-провайдера и музыкального бэкенда) рендерится внизу каждой вкладки через `<Suspense fallback={...}><SettingsScreen /></Suspense>` (строка 593-595). Это вне условного рендеринга табов, поэтому панель дублируется на всех девяти вкладках. API-эндпоинты (`GET /api/admin/settings`, `POST /api/admin/settings/provider`, `POST /api/admin/settings/backend`) остаются без изменений — меняется только клиентский рендеринг.

Дизайн-система: Liquid Glass (glass.css), компоненты `GlassPanel`, `Segmented`.

## Goals / Non-Goals

**Goals:**
- Удалить дублирование `SettingsScreen` из нижней части каждой вкладки админки.
- Перенести выбор AI-провайдера и музыкального бэкенда в компактную верхнюю панель над табами.
- Панель рендерится один раз и видна на всех вкладках.

**Non-Goals:**
- Изменение API (все эндпоинты и модель данных остаются теми же).
- Добавление новых настроек или контролов (только перенос существующих).
- Изменение поведения SettingsScreen в Telegram-боте (server/bot/admin-panel.ts не трогаем).

## Decisions

1. **Inline-рендеринг без lazy import.** Верхняя панель достаточно мала — два сегментед-контрола — lazy-загрузка не нужна. Логика `SettingsScreen` будет перенесена непосредственно в `AdminScreen` как отдельный компонент `AdminSettingsBar`. Исходный `SettingsScreen.tsx` может быть удалён (больше не используется) или оставлен для обратной совместимости — удаляем, чтобы не было мёртвого кода.

2. **Один API-запрос `GET /api/admin/settings` при монтировании admin-экрана.** Тот же вызов, что сейчас делает `SettingsScreen`. Данные прокидываются в `AdminSettingsBar` через пропсы или useState в AdminScreen. Кэширование не требуется — при переключении табов не нужно перезапрашивать.

3. **Состояние settings поднимается в AdminScreen.** Сейчас каждый `SettingsScreen` сам управляет своим состоянием. После переноса `useState<AdminSettings>` переезжает в `AdminScreen`, `AdminSettingsBar` получает его через пропсы. Это единственный source of truth для всей админки.

4. **Стилизация: компактный GlassPanel без h1.** Верхняя панель не нуждается в заголовке «Настройки администратора» — контролы говорят сами за себя. Используем `GlassPanel` без `className="reveal"` (без анимации появления, так как панель всегда видна) с минимальным внутренним отступом. Два сегментед-контрола в ряд (flex row) с подписями «ИИ» и «Музыка».

5. **Удаление `<Suspense>` обёртки.** После переноса настроек наверх она больше не нужна; статический контент табов грузится без отложенной секции.

## Risks / Trade-offs

- [Состояние провайдера/бэкенда может устареть, если админ не перезагрузит страницу] → Панель делает запрос при монтировании админ-экрана (то есть при переходе на экран Admin). Если админ меняет провайдера через бота, изменения подтянутся при следующем входе в админку. Для текущего UX это приемлемо.
- [Удаление SettingsScreen.tsx может быть BREAKING, если на него есть другие референсы] → Проверить imports в проекте. Если SettingsScreen нигде не импортируется, удаляем. Если есть другие референсы (например, в App.tsx или роутинге), перенаправить на Admin.
- [Компактный дизайн двух контролов в ряд может сломаться на узких экранах] → Использовать `flex-wrap: wrap` с отступами, чтобы на узких экранах контролы уходили в колонку.

## Migration Plan

Деплой стандартный (`deploy/deploy.sh`): только клиентские изменения. Откат — предыдущий билд мини-аппа без изменений. API не меняется.

## Open Questions

- Нужно ли сохранить `SettingsScreen.tsx` для возможного использования в будущем? Решение: удалить — мёртвый код не должен накапливаться, легко восстановить из git.
