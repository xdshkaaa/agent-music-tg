## Context

Сейчас `hasAccess(db, chatId)` в `server/access/entitlements.ts` проверяет только credits и subscription. Администраторы, чей chat_id в allowlist с флагом `is_admin=1`, не имеют льгот — paywall применяется ко всем одинаково.

Флаг `is_admin` уже хранится в таблице `allowlist` и проверяется через `getChatRole(db, chatId).isAdmin` в `server/lib/access-control.ts`. API middleware (`requireAdmin`) уже на него полагается.

## Goals / Non-Goals

**Goals:**
- Администраторы генерируют плейлисты бесплатно, без credits/подписки
- Обычные пользователи продолжают проходить paywall как раньше
- Минимальное изменение кода — одно место

**Non-Goals:**
- Не добавлять UI для управления льготами
- Не менять схему БД
- Не трогать логику consumption для обычных пользователей

## Decisions

- **Решение:** модифицировать `hasAccess()` — добавить проверку `isAdmin(db, chatId)`. Если админ — сразу `true`.
- **Почему не в run-generation.ts:** paywall-логика сконцентрирована в entitlements.ts; рассыпать проверки по run-generation.ts и API-роутам — хуже.
- **`consumeAccess()` тоже меняется:** админам не нужно списывать кредиты (которых у них может не быть). Делаем ранний `return` для админов.
- **Зависимость:** `entitlements.ts` сейчас не импортирует `access-control.ts`. Нужно добавить импорт `isAdmin`.

## Risks / Trade-offs

- **[Low] Админ с 0 credits не видит разницы с обычным пользователем в UI профиля** — credits=0 так и показывается, но generate работает. Не баг.
- **[Low] Если админ — владелец магазина, он может генерировать бесплатно бесконечно** — это ожидаемое поведение для разработки/демо.
- **[Low] `isAdmin()` делает лишний SQL-запрос** — при текущей нагрузке (1 админ) это не имеет значения.
