## Context

Текущий `deploy/deploy.sh` — линейный bash-скрипт (66 строк). Каждый шаг зависит от предыдущего, `set -e` прерывает выполнение при любой ошибке. Хардкод параметров (хост, пути), нет pre-flight проверок, нет автоматического отката, релизы копятся на VPS бесконечно.

Скрипт запускается с локальной машины разработчика. VPS доступен только по SSH (root). Технологии на сервере: Bun, systemd, Caddy, Cloudflare Tunnel.

## Goals / Non-Goals

**Goals:**
- Внешний конфиг для параметров деплоя (хост, пути, SSH_OPTS)
- Pre-flight pipeline: чистота git, ветка, typecheck, SSH connectivity, наличие .env на VPS
- Release name с git commit SHA для traceability
- Health check с автоматическим откатом `current` на предыдущий релиз при фейле
- Очистка N последних релизов после успешного деплоя
- Telegram-уведомление админу об успехе/провале
- Флаги: `--dry-run`, `--no-typecheck`, `--dirty`

**Non-Goals:**
- CI/CD интеграция (скрипт остаётся локальным)
- Zero-downtime deploy (Bun обрабатывает запросы синхронно, перезапуск сбрасывает in-flight)
- Поддержка нескольких таргетов за один запуск
- Ротация логов на VPS

## Decisions

**1. Config file over env vars**
- Формат: shell-совместимый `deploy.conf` с `source` в скрипте
- Дефолты через `${VAR:=default}`, конфиг поверх
- Файл в `.gitignore` — не коммитится
- Почему не env vars: скрипт запускается руками, не через CI — удобнее иметь файл рядом, чем экспортировать каждый раз

**2. Release name = `date +%Y%m%d-%H%M%S` + `git rev-parse --short HEAD`**
- Формат: `20250714-171509-a1b2c3d`
- Сортировка по алфавиту = сортировка по времени
- По SHA сразу видно, какой коммит задеплоен

**3. Auto-rollback: ssh одной сессией**
- Чтение предыдущего релиза и переключение symlink — одной ssh-командой, чтобы избежать race condition между двумя ssh-вызовами:
  ```bash
  ssh ... "cd $API_DIR && PREV=\$(ls -d releases/*/ | sort | tail -n 2 | head -n 1) && ln -sfn ..."
  ```

**4. Telegram-уведомления через Telegram Bot API**
- TELEGRAM_BOT_TOKEN читается из `.env` в корне проекта
- ADMIN_CHAT_IDS — первый ID из списка админов
- HTTP POST на `api.telegram.org/bot<token>/sendMessage` — не требует дополнительных зависимостей (curl уже используется)

**5. Dry-run: no-op режим**
- При `--dry-run` пропускать реальные rsync и ssh-команды, только выводить, что было бы сделано
- Pre-flight всё равно выполняется (чтобы проверить готовность)

**6. SSH одна команда на группу операций**
- Группировать связанные операции внутри одного `ssh "$HOST" "..."` чтобы снизить число TCP-соединений (соединение lossy, как сказано в комменте текущего скрипта)

## Risks / Trade-offs

- **Auto-rollback перезаписывает симлинк до проверки.** Если health check упал из-за проблем с .env или базой, откат на предыдущий релиз с теми же настройками не поможет. → `--force` флаг для пропуска rollback
- **Telegram токен в `.env`.** Если кто-то запустит скрипт и `.env` содержит токен для прода, возможна утечка. → `.env` уже в `.gitignore`, скрипт читает токен только для отправки сообщения, не логирует
- **Cleanup работает только при успешном деплое.** Если деплоишь 10 раз подряд с фейлом health check — релизы не чистит. → Можно добавить cleanup при rollback как улучшение
- **Региональность таймстампов.** `date` использует локальный часовой пояс — на машине в UTC и MSK будут разные форматы. → Фиксированный `TZ=UTC date ...`
