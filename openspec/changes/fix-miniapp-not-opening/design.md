# Design: fix-miniapp-not-opening

## Context

Прод-инцидент: https://miniapp.xdshka.party/ отдаёт 404. Диагностика на VPS (YOUR_VPS_IP):

- `systemctl is-active agent-music-tg` → active; `curl http://127.0.0.1:8787/healthz` → 200. API живой.
- `readlink /srv/www/miniapp.xdshka.party/current` → `releases/20260714-105705-b5adc42` — **директории не существует**. То же для `/opt/agent-music-tg/current`.
- В `releases/` обеих директорий лежат только 5 директорий старого формата имён (`20260714164503` … `20260714172446`).

Причина: `deploy/deploy.sh` сменил формат имени релиза на `YYYYMMDD-HHMMSS-<sha>` (spec `deploy-release`), но cleanup (строки 167–168) и rollback (строка 179) выбирают релизы через `ls | head/tail` — лексикографически. Дефис (`0x2D`) < цифра (`0x31`), поэтому `20260714-105705-b5adc42` сортируется *раньше* `20260714164503` и был удалён prune'ом как «самый старый» сразу после успешного деплоя. API-процесс пережил удаление (уже запущен, fd открыты), статика исчезла → 404. Health check прошёл до prune и статику не проверял.

## Goals / Non-Goals

**Goals:**
- Восстановить прод (Mini App открывается).
- Cleanup и rollback устойчивы к смешанным форматам имён релизов: порядок по mtime.
- Cleanup никогда не удаляет таргет `current`.
- Health check ловит отсутствие статики (висячий `current`), а не только API.

**Non-Goals:**
- Изменения кода приложения (server/, miniapp/).
- Переименование существующих старых релизных директорий (уйдут через prune естественно).
- Переезд на другой инструмент деплоя.

## Decisions

1. **Сортировка по mtime через `ls -1dt`** (новые первыми), «на удаление» = `tail -n +$((KEEP_RELEASES+1))`. Альтернатива — `sort -t- -k...` по имени с нормализацией форматов: хрупко, два формата парсить незачем; mtime корректен для обоих. `find -printf` недоступен одинаково везде — `ls -dt` проще и уже используется идиоматика ls в скрипте.
2. **Safeguard `current`**: перед `rm -rf` отфильтровать `$(readlink -f "$DIR/current")` из списка кандидатов (`grep -vxF`). Дешёвая страховка, ловит любые будущие ошибки сортировки — именно она предотвратила бы этот инцидент.
3. **Rollback выбор предыдущего**: `ls -1dt` по mtime, исключить `$RELEASE`, взять первый. Альтернатива «второй по mtime» без исключения — ломается, если новый релиз не самый свежий по mtime (rsync-таймстампы); явное исключение надёжнее.
4. **Health check статики**: `test -f "$STATIC_DIR/current/dist/index.html"` в той же remote-команде, что и curl `/healthz`. Альтернатива — curl через Caddy на `:8094`: тянет зависимость от Caddy-конфига; прямая файловая проверка точнее указывает на причину.
5. **Восстановление прода — redeploy фиксированным скриптом**, не ручная правка симлинков: старые релизы содержат устаревший код (последний коммит b5adc42 в них не попадал), а redeploy сразу проверяет фикс. Сначала фикс скрипта, потом deploy — иначе prune снова убьёт новый релиз (5 старых директорий всё ещё лексикографически «новее»).

## Risks / Trade-offs

- [SSH на VPS флаки под нагрузкой] → deploy.sh уже имеет timeout/retry; перед деплоем probe-подключение.
- [`ls -dt` на файле-не-директории в releases/] → шаблон `releases/*/` отбирает только директории; мусорные файлы не попадают.
- [mtime директории может обновиться при touch/bun install] → неважно: порядок нужен лишь относительный «новее/старее», а prune исключает `current` в любом случае.
- [Пока фикс не задеплоен, любой запуск старого deploy.sh повторит инцидент] → фикс первым шагом, деплой только после.

## Migration Plan

1. Фикс `deploy/deploy.sh` (cleanup, rollback, health check).
2. `./deploy/deploy.sh --dry-run` — pre-flight.
3. `./deploy/deploy.sh` (возможно `--dirty`, если рабочее дерево грязное) — создаёт новый релиз, чинит оба `current`, prune по mtime вычищает старые директории со временем.
4. Проверка: `curl -I https://miniapp.xdshka.party/` → 200; открыть Mini App в Telegram.

Rollback фикса: скрипт — обычный git revert; прод уже сломан, хуже не станет.
