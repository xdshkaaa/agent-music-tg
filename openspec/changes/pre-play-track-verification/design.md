## Context

Сейчас пользователь получает плейлист и видит все треки одинаково — без информации о том, рабочие они или нет. Проверка происходит только в момент воспроизведения (`GET /api/stream/:uri` запускает `yt-dlp`, который может упасть). Если трек удалён с YouTube, заблокирован по гео или имеет возрастные ограничения — пользователь узнаёт об этом только после нажатия «плей».

**Текущий flow:**
1. Генерация плейлиста → поиск треков через API YouTube Music/SoundCloud
2. Возврат треков с URI вида `ytm:videoId`
3. Пользователь нажимает «плей» → `yt-dlp` скачивает аудио → стриминг
4. Если `yt-dlp` падает — ошибка пользователю

Проблема: шаг 3 — единственная точка проверки, и она происходит по требованию пользователя.

## Goals / Non-Goals

**Goals:**
- После генерации плейлиста запускать фоновую проверку доступности каждого трека
- Показывать статус трека (проверяется/доступен/недоступен) в Mini App до нажатия «плей»
- Помечать битые треки в Telegram bot (при авторассылке аудио)
- Использовать лёгкий probe (без полной загрузки аудио)

**Non-Goals:**
- Не делать периодическую перепроверку треков
- Не менять потоковый playback (`/api/stream`) — он остаётся fallback-проверкой
- Не добавлять внешние сервисы/зависимости
- Не затрагивать логику поиска треков агентом

## Decisions

### 1. `Extractor.probe()` — быстрая проверка трека без скачивания аудио

Добавить метод `probe(uri)` в интерфейс `Extractor`, который проверяет, доступен ли трек для скачивания. Для `YtDlpExtractor` — использовать `yt-dlp --dump-json --no-playlist <url>`: это возвращает метаданные в JSON (поле `"availability"`, `"success"` и т.д.) без скачивания аудио-файла. Если yt-dlp возвращает `exit != 0` или в JSON есть признаки недоступности (geo-restricted, private, deleted) — трек помечается как битый.

**Альтернативы:**
- `yt-dlp --simulate` — тоже без скачивания, но дольше (медленнее `--dump-json`)
- `yt-dlp -g` (get URL) — быстрее всего, но не даёт метаданных о доступности; если URL получен — трек может быть всё ещё недоступен из-за geo/age
- **Решение**: `--dump-json` — оптимальный баланс скорости и информации

### 2. Статус проверки трека

Добавить тип `TrackVerificationStatus`:
```ts
type TrackVerificationStatus = "pending" | "verified" | "unavailable" | "checking";
```

Хранить статус в `FinalizedPlaylist` как поле `verification: Record<string, TrackVerificationStatus>` (ключ — `track.uri`). Это позволяет:
- Не менять базу данных — статус живёт в памяти сессии
- Просто сериализовать в JSON для Mini App

### 3. Запуск проверки

После успешной генерации (`startGeneration` / `resumeGeneration`) запускать `verifyTracks(tracks, extractor)` как fire-and-forget. Результаты записываются в тот же объект `playlist.verification` (по ссылке).

Mini App опрашивает статус через `GET /api/generate/status` или получает обновлённый список треков через новый endpoint `GET /api/generate/verification?id=<generation_id>`.

**Выбрано:** polling через `GET /api/generate/verification?ids=<uri1>,<uri2>` — простой endpoint, возвращает `Record<uri, TrackVerificationStatus>`. Mini App опрашивает раз в 2 секунды, пока все треки не будут проверены.

### 4. Отображение в Mini App

В `ResultsScreen.tsx` рядом с каждым треком добавить иконку:
- `checking` (проверяется) — спиннер `CircleNotch`
- `verified` (доступен) — зелёная галочка `CheckCircle`
- `unavailable` (недоступен) — красный `WarningCircle`
- `pending` — серая иконка (ожидание), без статуса до проверки

Клик на недоступный трек: показывать toast "Трек недоступен", не пытаться воспроизвести.

### 5. Telegram bot: авто-доставка

В `deliverAutoAudio()` / `deliverTrack()` — добавить проверку статуса. Если `track.uri` помечен как `unavailable` — пропустить трек, не пытаться скачать. Учесть что к моменту доставки проверка может ещё не завершиться (статус `pending` / `checking`).

### 6. Concurrency проверки

Проверять треки параллельно (до 3 одновременных probe-запросов). Использовать простой `pLimit` или `Promise.allSettled` с чанками по 3.

## Risks / Trade-offs

- **[Ложно-положительные]** `yt-dlp --dump-json` может вернуть ошибку из-за rate-limiting, а не из-за недоступности трека → **Mitigation**: не помечать как `unavailable` при первой ошибке — сделать 2 попытки с задержкой 1 сек
- **[Задержка генерации]** Проверка треков может замедлить "ощущение" генерации, если показывать её как обязательный шаг → **Mitigation**: проверка fire-and-forget, пользователь видит плейлист сразу, статусы приходят постепенно
- **[SoundCloud client_id]** Если `client_id` протух, probe может ложно падать → **Mitigation**: в `YtDlpExtractor` уже есть fallback (re-scrape), probe просто пробросит ошибку
