## Context

История загрузок в мини-приложении (`ProfileScreen` → вкладка «Загрузки») читает `GET /api/downloads`, который возвращает `listDownloads(db, chatId)` — записи таблицы `downloads`, принадлежащие `chat_id` пользователя. Записи туда попадают только из `server/api/audio-routes.ts` (`startJob` → `insertDownload` + `processDownload`) при нажатии «Скачать» в мини-приложении.

Однако большинство пользователей получают аудио иначе: они генерируют плейлист текстом в боте, и после генерации сервер вызывает `deliverAutoAudio` (`server/bot/auto-audio.ts`). Эта функция доставляет треки напрямую через `deliverTrack` и отправляет сводку, но **не создаёт запись `downloads`**. В итоге история в мини-приложении пуста для всех, кто качает через бота.

Таблица `downloads` уже существует, миграция идемпотентна (`CREATE TABLE IF NOT EXISTS`), `chat_id` фильтруется везде owner-scoped. Механизм записи и статусов уже готов в `downloads-store.ts` и `deliver.ts`.

## Goals / Non-Goals

**Goals:**
- Бот-автодоставка (`deliverAutoAudio`) создаёт запись `downloads`, видимую в «Загрузки» мини-приложения под тем же `chat_id`.
- Сохранить текущее поведение: пропуск `unavailable` треков, кеш `file_id`, сводное сообщение в чат, переносимость ошибок отдельных треков.
- Запись получает корректный итоговый статус (`done` / `partial` / `failed`) и список треков с per-track статусами.

**Non-Goals:**
- Не менять публичные API мини-приложения и эндпоинты `/downloads*`.
- Не менять логику повторной отправки («Скачать ещё раз») и удаления из истории — они универсальны.
- Не переписывать `processDownload`; бот использует свой цикл доставки (из-за пропуска `unavailable`), но переиспользует те же примитивы записи.

## Decisions

1. **Переиспользовать `downloads-store` вместо отдельной таблицы.** `deliverAutoAudio` вызывает `insertDownload(db, chatId, playlistName, tracks)` в начале, получает `DownloadRecord`, затем после доставки каждого трека — `setDownloadTracks(db, record.id, tracks)`, а в конце — `setDownloadStatus(db, record.id, finalStatusFor(tracks))`. Альтернатива (отдельная таблица бот-загрузок) отвергнута: дублирует логику и ломает универсальный список/повтор/удаление в мини-приложении.

2. **Сохранить пропуск `unavailable` треков.** Цикл в `deliverAutoAudio` оставляет `verificationStore.waitForFinal` и помечает пропущенные треки `status: "failed"` с `error: "недоступен"` (или просто не доставляет и ставит failed), чтобы итоговый статус и сводка отражали реальность. Это ближе к текущему поведению, чем универсальный `processDownload`, который бы попытался извлечь недоступный трек через yt-dlp.

3. **Сводное сообщение** формируется существующей `summaryText`-логикой уже в `deliverAutoAudio` — оставляем как есть, но считаем failed/skipped корректно.

4. **Сигнатура `deliverAutoAudio` не меняется** (`db, chatId, tracks, playlistName, api`), вызовы в `bot/index.ts` и `bot/generate.ts` остаются без правок.

## Risks / Trade-offs

- [Risk] Запись `downloads` создаётся до начала доставки со статусом (по умолчанию `pending`) — если доставка упадёт целиком, запись останется `pending`/`failed`. → Mitigation: как и в мини-приложении, `setDownloadStatus` вызывается в конце; при исключении вне цикла ловим и ставим `failed`, чтобы история не «висела» вечно. Уже существует поведение `hasActiveDownload`, учитывающее `pending`/`processing`.
- [Risk] Дублирование: если пользователь и в боте, и через «Скачать» в мини-приложении качает один плейлист — будут две записи. → Приемлемо: это две реальные доставки, история отражает факт.
- [Trade-off] Бот-загрузки не проходят через `processDownload`, поэтому нет единого места доставки. → Сознательно ради сохранения пропуска `unavailable`; логика доставки трека (`deliverTrack`) переиспользуется без дублирования.

## Migration Plan

- Изменение только серверной логики `deliverAutoAudio`; БД-миграция не требуется (таблица есть).
- Деплой штатный (`deploy/deploy.sh`): перезапуск systemd-юнита. Старые записи сохраняются; новые бот-загрузки начнут появляться в истории сразу.
- Откат: предыдущий релиз просто не пишет записи для бота — данные не теряются, история лишь перестанет пополняться из бота.
