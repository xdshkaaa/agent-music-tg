## Context

Сейчас аватарка пользователя сохраняется при `/start` через `getUserProfilePhotos` (берётся largest `file_id`), а на `GET /api/me` сервер дергает `getFile` и возвращает прямой URL на Telegram CDN. Если у пользователя видео-аватарка (анимированный профиль), `getFile` может вернуть `file_path` с расширением `.mp4`. Браузер не воспроизведёт MP4 в `<img>`, а в боте `replyWithPhoto` упадёт с ошибкой.

Нужно детектить такие случаи и конвертировать в статичный JPEG.

## Goals / Non-Goals

**Goals:**
- Детекция анимированных аватарок (MP4, GIF) при построении `photoUrl` в `/api/me`
- Конвертация первого кадра видео/GIF в JPEG через ffmpeg
- Кеширование сконвертированного JPEG, чтобы не конвертировать при каждом запросе
- Graceful fallback: если конвертация не удалась — placeholder, никаких 500 ошибок
- В боте `/profile`: fallback на текстовый профиль при неудачной отправке фото

**Non-Goals:**
- Поддержка анимированных аватарок в Mini App (не показываем видео, только статику)
- Перезапись аватарки при каждом визите — только если пользователь перезапустил `/start`
- Проксирование аватарок через наш сервер (используем прямые URL Telegram, но для сконвертированных — локальные)

## Decisions

1. **Детекция по расширению из `getFile`** — проще всего: после `getFile` проверяем `file_path` на `.mp4`, `.mov`, `.gif`. Если ни одно не совпало — отдаём оригинальный Telegram URL как сейчас. Альтернатива (MIME-тип через `fetch` HEAD) — лишний запрос.

2. **ffmpeg для конвертации** — уже установлен на VPS через `deploy.sh`. Команда: `ffmpeg -i <input> -vframes 1 -q:v 2 <output>.jpg`. Альтернатива (`sharp`) не умеет видео, `fluent-ffmpeg` — лишняя зависимость; используем `child_process.exec`.

3. **Кеширование по `file_unique_id`** — `file_unique_id` не меняется для одного и того же файла в отличие от `file_id`. Храним сконвертированные JPEG в `data/avatars/<file_unique_id>.jpg`. При повторном запросе проверяем наличие файла на диске.

4. **Новый модуль `server/avatar.ts`** — выделяем логику конвертации и детекции в отдельный файл, не засоряем routes.ts.

5. **Обработка в боте `/profile`** — `replyWithPhoto` кидает ошибку для видео-файлов. Ловим её и отправляем только текст.

## Risks / Trade-offs

- [ffmpeg может не быть в dev-окружении] → проверять наличие `ffmpeg` перед вызовом, если нет — пропускать конвертацию, отдавать placeholder
- [Дисковое кеширование со временем] → не критично для `data/avatars/`, файлы мелкие; очистка не нужна
- [Телеграм меняет `file_unique_id` при смене аватарки] → это нормально, старый файл останется мусором; объём ничтожный
- [Конвертация каждого кадра при каждом `/api/me`] → только при первом запросе, потом кеш
