# Design: download-audio-history-playback

## Context

Playlists are generated by an agent loop against resolve-only music backends (`ytm:<videoId>` / `sc:<id>` URIs with deep links; see `server/music/types.ts`). Results are shown in the Mini App (`ResultsScreen`) and sent to chat as text (`formatPlaylistReply`). There is no audio anywhere: no downloads, no playback. The bot runs on a VPS (Bun + grammY long-polling + Hono API behind Cloudflare Tunnel), data in `bun:sqlite`.

Constraints:
- Bot API `sendAudio` upload limit is 50 MB per file; uploads from the VPS are slow, so repeat sends must reuse Telegram `file_id`.
- The VPS has limited disk; downloaded files are temporary, not a media library.
- The Mini App authenticates via Telegram initData (existing middleware) — the streaming endpoint must not be an open proxy.
- YouTube extraction requires `yt-dlp` + `ffmpeg` as system dependencies.

## Goals / Non-Goals

**Goals:**
- One-tap «Скачать» on results: playlist tracks arrive in the user's bot chat as audio messages.
- Per-user downloads history with re-send and delete, surfaced in the profile area.
- Inline playback in the Mini App via a server streaming endpoint.
- Idempotent, cheap repeats: `file_id` cache means a re-download never re-extracts or re-uploads.

**Non-Goals:**
- No permanent audio storage on the VPS (cache is Telegram-side via `file_id`; local files are deleted after upload).
- No download of arbitrary user-supplied URLs — only track URIs produced by this app's generations.
- No offline/background playback, queue management, or full player UI beyond play/pause/progress.
- No per-download billing changes (downloads consume no credits in this change).

## Decisions

### D1: Extraction via `yt-dlp` subprocess, one wrapper for both backends
`yt-dlp` handles both YouTube Music (`https://music.youtube.com/watch?v=<id>`) and SoundCloud (`https://api.soundcloud.com/tracks/<id>` or permalink) URLs. A single `server/audio/extractor.ts` maps `Track.uri` → source URL → `yt-dlp -x --audio-format mp3` into a scratch dir. Alternative considered: per-backend Node libraries (ytdl-core etc.) — rejected: fragile against site changes, no SoundCloud parity; `yt-dlp` is the maintained standard and already fits the VPS deploy model (installed via deploy script).

### D2: Telegram `file_id` cache as the canonical audio cache
New `audio_cache` table: `uri PRIMARY KEY, tg_file_id, title, artist, duration_ms, size_bytes, created_at`. Flow: on send, hit cache → `sendAudio(file_id)`; miss → extract, upload once (to the requesting chat), store returned `file_id`, delete local file. Alternative: keep MP3s on disk — rejected: disk pressure, redundant with Telegram's free storage. Consequence: streaming (D5) cannot reuse `file_id` directly and keeps its own short-lived disk cache.

### D3: Download job runs async server-side, delivery to chat via bot
`POST /api/download` (body: playlist name + tracks) validates access, inserts a `downloads` row with per-track statuses, returns `202 { downloadId }` immediately, then processes tracks sequentially in the background: extract → sendAudio → update track status. Per-track failures (region lock, >50 MB, extraction error) mark that track `failed` and continue; the bot sends a summary message at the end. Sequential (concurrency 1 per job, one job per user at a time) protects the VPS CPU/disk. Alternative: synchronous request — rejected: a 15-track playlist takes minutes, HTTP timeout guaranteed.

### D4: History schema — playlist-level rows with embedded track list
`downloads` table: `id, chat_id, playlist_name, tracks_json (uri/title/artist/status per track), status ('pending'|'processing'|'done'|'partial'|'failed'), created_at`. JSON column instead of a normalized `download_tracks` table because tracks are read/written only as a unit and the app already stores structured session state as JSON (`sessions.state`). Delete removes the history row only — cached `file_id`s in `audio_cache` stay (they are user-independent).

### D5: Streaming endpoint with short-lived local cache
`GET /api/stream/:uri` (initData-authed, same middleware as other API routes): if a fresh local file exists in the stream cache dir, serve it with `Content-Type: audio/mpeg` and Range support; otherwise extract via D1 into the cache dir, then serve. Cache is LRU-capped (e.g. 200 MB / 24 h TTL, swept on access). URI must match `^(ytm|sc):[\w-]+$` — no arbitrary URLs. Alternative: proxy-pipe yt-dlp stdout — rejected: no Range support (seek broken in `<audio>`), repeated extraction per play.

### D6: Mini App UI placement
- `ResultsScreen`: «Скачать» button beside «Новый плейлист» (`DownloadCloud` icon); tap → `api.download(playlist)` → button becomes confirmation state («Отправляю в чат…» → «Отправлено в чат ✓»). Each track row also gets a play/pause control wired to the shared player.
- Profile: `ProfileScreen` gains a `Segmented` switch (existing component) — «Покупки» | «Загрузки». Downloads list shows playlist name, date, track count, status; actions: re-send to chat (`POST /api/downloads/:id/resend`) and delete (`DELETE /api/downloads/:id`, with confirm).
- Player: single global `<audio>` element in a `PlayerBar` component rendered by `App` above the dock; state (current uri, playing, progress) in a small context/store. One track at a time.

Alternative for downloads UI: a fourth dock tab — rejected: dock already has 3–4 tabs (admin conditional); user asked for it "in profile in tab bar", the segmented control inside profile satisfies that without crowding the dock.

## Risks / Trade-offs

- [yt-dlp breakage: YouTube changes format/signatures] → pin + auto-update `yt-dlp` in deploy script; per-track failure tolerance means one broken extractor never kills a whole playlist; error surfaced in chat summary.
- [Tracks > 50 MB Bot API limit] → `yt-dlp` audio-only at ~128–192 kbps mp3 keeps typical songs at 3–10 MB; skip and mark `failed` with a reason if the file still exceeds the limit.
- [VPS CPU/disk pressure from concurrent extractions] → one active job per user, global concurrency cap (2 extractions), scratch files deleted immediately after upload, stream cache LRU-capped.
- [`file_id` invalidation (Telegram can expire them)] → on `sendAudio(file_id)` failure, fall back to re-extract + re-upload and refresh the cache row.
- [ToS: downloading from YouTube/SoundCloud violates their terms] → accepted by the project owner; scope limited to tracks the app itself resolved, no public download API.
- [Streaming endpoint cost: first play blocks on extraction] → UI shows a loading state on the play button; stream cache makes subsequent plays instant.

## Migration Plan

1. Deploy script: install `yt-dlp` + `ffmpeg` on the VPS (idempotent), create scratch/stream-cache dirs.
2. DB migration is additive (`downloads`, `audio_cache` tables in `migrate()`), safe on existing data.
3. Ship server + miniapp together; feature is additive, no rollback hazard — rolling back the binary leaves unused tables behind, which is harmless.

## Open Questions

- Should downloads consume credits or be gated by subscription? (Current answer: free for anyone with access; revisit if VPS load becomes a problem.)
- SoundCloud extraction may require a client_id for some tracks — verify `yt-dlp` handles the tracks the SoundCloud backend resolves; if not, restrict download to `ytm:` URIs initially and mark `sc:` tracks unsupported.
