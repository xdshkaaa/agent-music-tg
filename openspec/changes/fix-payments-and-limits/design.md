# Design

## Context

Инфраструктура платежей уже существует: офферы (`offers-store`), инвойсы (`invoices-store`), Crypto Pay (`crypto-pay.ts`, `webhook.ts`, `poller.ts`), Telegram Stars (`stars.ts`), идемпотентное исполнение (`fulfillment.ts`), кошелёк пользователя (`users-store`: `credits`, `trialCredits`, `subscriptionUntil`), проверка/списание доступа (`entitlements.ts`), вызовы из `run-generation.ts`. Но цепочка целиком не верифицирована как рабочая, а подписка безлимитна.

Ключевые наблюдения по текущему коду:
- `consumeAccess` вызывается после успешной генерации во всех трёх функциях (`startGeneration`, `resumeGeneration`, `extendGeneration`) — clarify и ошибки бесплатны. Это соответствует желаемому поведению; аудит должен подтвердить, что стрим-роуты (`/generate/stream` и т.п.) идут через эти функции.
- `fulfillInvoice` и `fulfillStarsPayment` идемпотентны через переход pending→paid / INSERT OR IGNORE.
- `hasAccess`: payments off → все; админ → всегда; иначе credits>0 | активный trial | подписка.
- Rate limit по подписке отсутствует полностью.

## Goals / Non-Goals

**Goals:**
- Подтверждённый рабочий цикл покупки для обоих методов оплаты, с фиксами найденных разрывов.
- Гарантированное списание ровно одного кредита за успешную генерацию на всех путях.
- Корректная работа подписки (доступ, продление, истечение).
- Почасовой рейт-лимит для подписчиков с внятным сообщением об исчерпании.

**Non-Goals:**
- Новые методы оплаты, изменение модели офферов, рефанды.
- Рейт-лимиты для кредитных/trial-пользователей (они платят за каждую генерацию).
- Дневные/месячные квоты — только скользящий час.

## Decisions

### D1. Rate limit считается по `generations` таблице, без новой таблицы
Успешная генерация уже пишется в `generations` (`insertGeneration` / `appendTracksToGeneration`) с `created_at`. Лимит = `COUNT(*) WHERE chat_id = ? AND created_at > now - 3600`. Скользящее окно, ноль новой инфраструктуры, автоматически «только успешные» (clarify/ошибки строк не создают).
Альтернатива — отдельная таблица `rate_limit_events` — отвергнута: дублирует данные.
Нюанс: extend-генерация обновляет существующую строку (`appendTracksToGeneration`), а не вставляет новую. Для честного учёта нужно считать extend-запуски тоже — решение: добавить колонку-счётчик не будем; вместо этого extend при подписке проверяет тот же COUNT (небольшая недооценка допустима) — фиксируем это в спеке как «успешные генерации, создающие плейлист»; extend учитывается через новую лёгкую таблицу НЕ вводится. Компромисс задокументирован в Risks.

### D2. Лимит применяется только когда доступ идёт по подписке
Порядок в `hasAccess`/`consumeAccess` уже: credits → trial → subscription. Лимит проверяется в новой функции `checkSubscriptionRateLimit(db, chatId)` только если пользователь получил бы доступ именно по подписке (нет кредитов и нет активного trial). Кредитный пользователь с истёкшей подпиской не троттлится. Админы и режим `paymentsEnabled=false` — без лимита.

### D3. Новый исход `rate_limited` в `GenerationOutcome`
`{ status: "rate_limited"; retryAt: number }` (unix-секунды, когда самая старая генерация окна выйдет из часа). Отдельный статус, а не `needs_purchase`: UI должен показать «подождите до HH:MM», а не «купите». Проверка — в начале `startGeneration`/`resumeGeneration`/`extendGeneration`, до любого LLM-вызова. Пробрасывается в JSON-роуты (HTTP 429) и SSE (`frame: rate_limited`), в бота — текстом.

### D4. Конфиг `SUBSCRIPTION_HOURLY_LIMIT`
`server/env.ts`: целое, по умолчанию `25`, `0` = лимит отключён. Валидация как у остальных env-переменных.

### D5. Аудит покупки — тестами, не переписыванием
Цикл покупки чинится через интеграционные тесты, повторяющие полный путь: `POST /api/invoices` (crypto и stars) → эмуляция webhook/`successful_payment` → проверка баланса в `/api/me` и доступа в `hasAccess`. Найденные разрывы фиксируются точечно. Никаких превентивных рефакторингов работающего кода.

## Risks / Trade-offs

- [Extend-генерации не создают строку в `generations` → недоучёт в окне лимита] → Принято осознанно: extend редок, недоучёт консервативен в пользу пользователя; при злоупотреблении добавим счётчик отдельным изменением.
- [Webhook в проде может быть не настроен в Crypto Pay UI] → Поллер (`poller.ts`) уже является фолбэком; тест подтверждает fulfil через поллер.
- [Часовое окно по `created_at` зависит от часов сервера] → unix-время, один сервер — риск минимален.
- [429 в SSE-стриме: клиент уже открыл соединение] → отдаём кадр `rate_limited` и закрываем стрим, миниапп показывает сообщение.

## Migration Plan

1. Новая env-переменная с дефолтом — деплой без изменения `.env` безопасен (лимит 25/час включится для подписчиков сразу).
2. Схема БД не меняется — отката данных нет.
3. Rollback: обычный откат релиза; либо `SUBSCRIPTION_HOURLY_LIMIT=0` для отключения только лимита.

## Open Questions

- Значение лимита по умолчанию (принято 25/час) — админ может изменить через `.env`; выносить в админ-панель не будем в этом изменении.
